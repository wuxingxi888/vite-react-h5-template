# React 通用样式规范（评审用）

## 1. 总体原则（NON-NEGOTIABLE）

- UI 组件库以 `antd-mobile` 为准（交互与可访问性由组件库提供为主）
- 布局/间距/排版优先使用 `UnoCSS` 原子类（减少样式冲突、提升一致性）
- 需要工程化/复杂样式时使用 `SCSS Modules`，页面与组件局部样式文件统一优先 `index.module.scss`
- 暗黑模式统一走项目主题同步：`html` 上切换 `light/dark` 类，并通过 CSS 变量 `--adm-color-primary` 注入主题色
- 颜色相关规则遵循“可主题化优先”：可随主题变化的颜色禁止随意硬编码

> 参考：Ant Design Mobile CSS 变量文档  
> https://mobile.ant.design/zh/guide/css-variables

## 2. 主题色与 CSS 变量（NON-NEGOTIABLE）

### 2.1 Ant Design Mobile 变量约定

1. **主题色主变量**：项目应在统一主题同步入口向 `document.documentElement` 写入：
   - `--adm-color-primary`: 当前主题色（浅/暗色下保持一致策略）
2. **其他 antd-mobile 变量**：如需使用非主色的组件内部变量，优先查官方文档并使用对应 `--adm-*` 变量。

### 2.2 暗黑模式适配规则

- 暗黑/浅色切换来自项目对 `document.documentElement` 的统一处理：
  - `root.classList.add(themeMode)`：`light` 或 `dark`
  - `root.style.setProperty('--adm-color-primary', themeColor)`
- 在样式中：
  - UnoCSS：使用 `dark:(...)` 或 `dark:text-*` 形式
  - SCSS：用 `@include use-theme { ... }` 读取主题态变量（见下一节）
- **禁止**在组件/页面里自行实现 `prefers-color-scheme` 逻辑或重复 `setProperty` 主题色

### 2.3 项目主题（SCSS Mixins）使用规范

项目可在 `src/styles/` 下提供统一主题能力，例如：
- `@include use-theme { ... }`：根据 `html.light` / `html.dark` 生成主题分支
- `get-var('xxx')`：获取主题态颜色值（如 `main-text-color`、`main-bg-color`、`main-card-bg-color` 等）

在组件 SCSS 中，遵循以下写法：

```scss
.some-root {
  @include use-theme {
    color: get-var('main-text-color');
    background-color: get-var('main-card-bg-color');
  }
}
```

### 2.4 硬编码颜色的边界（允许项与禁止项）

禁止：
- 在“可主题化”的样式里硬编码主背景/主文字/强调色（例如会随浅/暗色改变的主视觉）

允许（需要写明理由或可被接受的例外）：
- 组件库默认样式无法覆盖、且该样式确定为“固定装饰性颜色/品牌色/演示色”
- 图标/插画/演示页的演示效果（例如 `FigmaDemo`、`UnoCss` 演示）
- 纯算法/中性色（例如灰阶抗锯齿边界）且不会破坏暗黑可读性

评审时建议用问题校验：
- 这处颜色在切换 `light/dark` 后是否仍保持足够对比度与语义一致？

## 3. UnoCSS 使用规范

### 3.1 使用范围

- 优先用于：布局、间距、尺寸、排版、flex/grid、文字溢出、颜色（尤其是暗黑变体）
- 复杂视觉效果/需要复用的样式：放到 `SCSS`（使用 `@apply` 或写入局部样式类）

### 3.2 选择约定

1. 数值尺寸优先使用“括号任意值”形式（与本项目示例一致）：
   - `mt-[30px]`、`text-[14px]`、`w-[120px]`
2. 暗黑模式用变体写法：
   - `dark:text-garyWhite`、`dark:(bg-gray-800 text-white)`
3. 动态/组装类：
   - 通过 `classList.add/remove` 等方式操作（在演示页/交互逻辑里可见）
4. 文本截断：
   - 多行省略按 `line-clamp-*` 使用

### 3.3 与 antd-mobile 的配合

- antd-mobile 组件内部结构复杂，优先用：
  - UnoCSS 控制外层布局与容器间距
  - SCSS 局部覆盖 `.adm-*` 内部样式（见第 4 节）

## 4. SCSS 编写规范（作用域与覆盖策略）

### 4.1 文件组织

模板建议的样式加载方式：
- 在入口文件统一引入 UnoCSS 与全局基础样式
- 每个组件/页面按需在 `index.tsx` 中引入 `./index.module.scss`

因此建议：
- `index.tsx` 与 `index.module.scss` 置于同一目录
- 样式入口类建议以根容器 `.<component-name>` 开头，降低冲突概率

### 4.2 选择器作用域（避免“污染全局”）

在组件/页面 SCSS 中：
- 禁止直接写 `body` / `html` / `#root` 等全局选择器
- 禁止使用过宽的通配符选择器（例如 `*`）
- 禁止跨组件嵌套到不可控层级（确保样式覆盖范围可推导）

### 4.3 覆盖 antd-mobile（必须可控）

推荐覆盖 antd-mobile 的方式是：
- 仅在局部根容器内嵌套 `.adm-*` 选择器
- 使用 `index.module.scss` 隔离到对应组件/页面

示例（结构化覆盖）：

```scss
.login-form {
  .adm-list {
    --border-inner: none !important;
  }

  .adm-switch {
    --height: 26px;
    --width: 43px;
  }
}
```

禁止：
- 在任意 SCSS 文件里全局重置 `.adm-*`（除非放在全局样式入口并经评审批准）
- 不要无脑 `!important`；若必须使用，需说明为什么（例如组件库样式权重无法覆盖）

### 4.4 适配安全区（safe area）

需要底部安全区时：
- 优先使用 `src/styles/` 下统一的 safe-area mixin 或工具类
- 避免在业务组件中散落硬编码安全区适配逻辑

## 5. 设计稿颜色提取与落地

### 5.1 常用字段与示例

| 属性 | 含义 | 示例 |
|------|------|------|
| `fill` | 背景色 | `"#13131c73"` |
| `stroke` | 边框色 | `"#5f50ffff"` |
| `opacity` | 透明度 | `0.5` |

### 5.2 透明色转换规范（必须）

设计稿常见 `#RRGGBBAA` 格式：
- `AA` 是透明度（十六进制）
- 转换为 `opacity = AA / 255`
- 例如 `"#13131c73"`：
  - `0x73 = 115`
  - `115 / 255 ≈ 0.45`

然后落到 CSS/SCSS：
```scss
.button {
  background-color: rgba(19, 19, 28, 0.45);
  backdrop-filter: blur(17.5px);
}
```

### 5.3 落地顺序建议

1. 先判断颜色是否需要随主题变化（切换 `light/dark` 语义是否会变）
2. 如需主题变化：
   - UnoCSS 用 `dark:*` 变体或使用项目主题 mixin（SCSS）
3. 否则再使用透明转换后的 `rgba`

## 6. 动画与过渡规范（样式侧）

- 简单过渡优先用 CSS/UnoCSS（如 `transition-*`）
- 页面/组件进入离开动画：
  - 统一使用项目已有的路由/动画类策略
- 对复杂动效与时序：
  - 交互/动画逻辑优先用 `framer-motion`（样式规则侧保持“只定义外观”）

## 7. 层级与 z-index

- z-index 必须有语义来源：导航、悬浮栏、遮罩、动画层等
- 避免使用无限制的 `9999`；在组件内部给出合理层级（本项目示例中：固定栏/悬浮层常用 `1000`）
- 动画路由进入的层级应由项目统一动画层策略控制，业务仅负责挂载与触发

## 8. 评审要点（PR 样式检查清单）

1. 是否优先用 `antd-mobile` 组件完成交互，而不是自定义“假组件”
2. 是否优先用 UnoCSS 控制布局/间距/排版，避免大段 SCSS 重写
3. 暗黑模式下视觉是否正确（尤其是文字对比度、按钮强调色、容器背景）
4. SCSS 作用域是否被控制在本组件根容器下，是否误影响其他页面
5. antd-mobile 样式覆盖是否仅限局部 `.adm-*`，且没有无理由全局覆盖
6. 设计稿颜色是否按透明转换规范落地（`#RRGGBBAA -> rgba`）
7. 是否出现主题色硬编码（如主背景/主文字/强调色），有无理由与替代方案

## 9. 模板边界说明

- 本规则描述的是 React 通用模板口径，不绑定某个具体项目文件名，如 `src/plugins/index.ts`、`useThemeSync`、`theme.scss`、`entry.scss`
- 若某个项目有自己的主题同步实现，可在项目扩展规则中补充“具体接线方式”，但不得反向覆盖本模板的样式基线
