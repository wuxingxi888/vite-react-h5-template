# 组件规范

## 组件结构

- 组件目录在 `src/components` 下，每个组件单独一个目录
- 组件目录名统一使用 `PascalCase`（如 `HomeList`、`ErrorBoundary`）
- 组件文件名为 `index.tsx`，样式文件为 `index.module.scss`
- 根级 `src/components/index.ts` 可作为对外公共出口，但不强制所有组件都集中导出
- 组件必须优先使用 SCSS Modules；全局样式仅允许放在 `src/styles/`
- 组件必须使用清晰的 TS 类型定义，Props 接口必须明确
- 使用 `classnames` 负责动态 class 管理

### ComponentName 命名约定

- 本文中的 `<ComponentName>` 指组件目录名与组件名，统一使用 `PascalCase`
- 推荐语义化命名，优先使用领域名词（如 `OrderCard`、`UserProfilePanel`）
- 避免使用泛化命名（如 `Item`、`Box`、`Wrapper`），除非该组件确实是通用基础布局组件

如需查看创建组件的完整示例与拆分步骤，请使用技能文件：

- `.agents/skills/create-component/SKILL.md`（若技能尚未提供，则以本规则为准）

## 组件层级规划

- 单个 `.tsx` 文件建议不超过 400 行（作为拆分的参考阈值）
- 遵循单一职责原则，每个组件应有明确的职责边界
- **页面级组件**：仅在当前页面使用 → 放在 `src/views/<page-name>/components/`
- **通用组件**：跨页面复用 → 提取到 `src/components/`

## 组件放置决策树

新增组件时，按以下流程判断放置位置：

```
组件需要在多个页面使用吗？
├─ YES → 通用组件 → 放在 `src/components/<ComponentName>/`
└─ NO → 仅在单页使用 → 放在 `src/views/<page-name>/components/<ComponentName>/`
```

**示例**：

- `ErrorBoundary`（多处使用）→ `src/components/ErrorBoundary/`
- `HomeHeader`（仅首页使用）→ `src/views/home/components/HomeHeader/`
- `CraftSheetCard`（仅工艺单页面使用）→ `src/views/craft-sheet/components/CraftSheetCard/`

**约束**：`src/components/` 中的组件必须是真正通用的，业务耦合组件禁止放入。

## 组件类型变更流程（人工迁移）

当组件后期发生“业务组件 <-> 通用组件”类型变化时，必须执行以下流程。

### 触发条件

- **业务组件晋升为通用组件**：同一组件被 2 个及以上页面稳定复用，且可抽离页面专属业务依赖
- **通用组件降级为业务组件**：组件长期仅服务单页面，且包含明显业务耦合逻辑

### 迁移步骤

1. 移动组件目录到目标层级：
   - 晋升：`src/views/<page-name>/components/<ComponentName>/` -> `src/components/<ComponentName>/`
   - 降级：`src/components/<ComponentName>/` -> `src/views/<page-name>/components/<ComponentName>/`
2. 更新导出：
   - 若项目存在 `src/components/index.ts` 公共出口，晋升后应按需补充导出
   - 降级后应移除对应公共导出，避免错误暴露页面私有组件
3. 全局修复导入路径，确保无旧路径残留
4. 执行类型检查/构建检查，确认编译通过
5. 对受影响页面做功能回归验证

### 兼容策略

- 允许短期保留兼容转发（re-export）以降低一次性改动风险
- 兼容转发最长保留 1 个迭代，下一迭代必须清理

### 禁止项

- 禁止“名为通用、实为业务”的组件长期留在 `src/components/`
- 禁止组件迁移后不更新导出与导入路径

## 最佳实践补充

- 不鼓励维护过大的根级 barrel export；通用组件较多时，应优先按领域或目录就近导出，避免形成单个超大入口
- 页面私有组件禁止通过根级公共出口暴露
