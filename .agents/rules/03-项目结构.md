# 项目结构（NON-NEGOTIABLE）

## 命名总则（统一口径）

- 本规则统一使用复数目录命名：`assets`、`components`、`constants`、`hooks`、`http`、`routes`、`stores`、`styles`、`types`、`utils`、`views`
- `src` 下目录名统一使用 kebab-case（如 `user-center`、`login`）
- 未在本规则声明用途的目录，视为非标准目录，禁止新增

## 标准目录结构

```
src/
├── assets/        # 字体、图标、静态图片等静态资源
├── components/    # 跨页面复用的通用 UI 组件
├── constants/     # 常量、枚举、配置常量
├── hooks/         # 复用型自定义 Hooks
├── http/          # API 请求封装（按业务模块）
├── routes/        # 路由注册、模块配置、守卫与公共路由工具
├── stores/        # Zustand 全局状态（按业务模块）
├── styles/        # 全局样式、主题、变量、工具样式
├── types/         # 共享类型声明（DTO、模型、公共类型）
├── utils/         # 与业务弱相关的通用工具函数
├── views/         # 页面级模块（路由最终渲染页面）
└── 根级文件        # App.tsx、main.tsx 等入口文件
```

## 结构约束（禁止在 src 下新建非标准目录）

| 类型 | 目录 | 放置内容 | 禁止内容 | 导出规范 |
|------|------|----------|----------|----------|
| 常量 | `src/constants/` | 业务常量、枚举、配置常量 | 与运行时状态耦合的逻辑 | 目录内提供 `index.ts` 聚合导出 |
| 状态 | `src/stores/` | Zustand store、actions、selectors | 直接编写视图 JSX | 对外暴露 `useXxxStore` |
| 工具 | `src/utils/` | 纯函数、无 UI 依赖工具 | 持有全局可变状态 | 目录内提供 `index.ts` 聚合导出 |
| 类型 | `src/types/` | 模型、DTO、响应体、共享类型 | 运行时代码与副作用逻辑 | 仅导出 type/interface/enum |
| 通用组件 | `src/components/` | 跨页面复用组件 | 仅单页面使用的组件 | 每个组件目录包含 `index.tsx`，可选 `index.ts` 转发导出 |
| 路由 | `src/routes/` | 路由模块、路由守卫、懒加载装配 | 页面 UI 具体实现 | 每个路由模块默认导出路由配置 |
| 页面 | `src/views/` | 页面容器、页面编排、页面私有逻辑 | 全局通用组件沉淀 | 页面目录默认 `index.tsx` 作为入口 |

## 页面内组件放置规则

页面专用组件放在 `src/views/<page-name>/components/` 目录：

```
src/views/<page-name>/
  ├─ components/           # 仅供当前页面使用的子组件
  │   └─ ComponentName/
  │       ├─ index.tsx
  │       └─ index.module.scss
  ├─ hooks/                # 仅供当前页面使用的 hooks（可选）
  ├─ index.tsx
  └─ index.module.scss
```

**判断标准**：

- 通用组件：多处复用、业务无关 → `src/components/`
- 页面级组件：仅单页使用 → `src/views/<page-name>/components/`

**约束**：`src/components/` 只存放通用组件，禁止将仅单页使用的组件放入。

## 组件边界与迁移引用

- 组件命名、通用组件与页面级组件的边界判断，以及“通用 <-> 业务”双向迁移流程，统一以 `04-组件规范.md` 为准
- 本文档负责目录职责边界，具体组件治理细则不得与 `04-组件规范.md` 冲突

## 类型组织规则（统一口径）

- `src/types/` 统一采用目录化组织：`src/types/<feature>/`
- 每个业务类型目录固定为：
  - `model.ts`：实体模型、领域对象类型
  - `api.ts`：请求/响应 DTO 类型
  - `index.ts`：对外统一导出
- 仅当某业务只包含单一类型且无 2 周内扩展计划时，可使用 `src/types/<feature>.ts` 单文件；一旦新增第二类类型，必须升级为目录结构

## 接口未完成阶段策略（同文件管理）

当后端接口未完成时：

### 目录结构

```
src/
  ├─ http/
  │   └─ <feature>.ts            # mock API 文件（含请求模拟）
  └─ types/
      └─ <feature>/              # 优先目录化；最小场景可单文件
          ├─ model.ts
          ├─ api.ts
          └─ index.ts
```

### 原则

1. 接口文件：创建 `src/http/<feature>.ts`，包含 mock 数据与模拟请求函数
2. 类型定义：优先使用 `src/types/<feature>/` 目录结构，保持与正式接口阶段一致
3. 状态管理：接口未完成阶段默认不新增全局 `stores` 模块；确需跨页面共享时方可创建，并在代码注释中说明原因
4. 页面目标：优先保证页面结构与交互流程可联调，其次完善边界分支

### 禁止

- 禁止在 mock 中引入超过 3 层的对象嵌套（数组项不计层级）
- 禁止为“未来可能用到”预留未消费字段；mock 字段应与当前页面消费字段一一对应
- 禁止在接口未完成阶段将 mock 常量散落到多个目录；统一收敛在 `src/http/<feature>.ts`

## 规则优先级

当“长期结构约束”与“接口未完成阶段策略”冲突时，按以下优先级执行：

1. 结构与命名总则（最高优先级）
2. 类型组织规则
3. 接口未完成阶段策略（仅在后端接口未就绪时生效）

## 模板说明

- 本模板固定使用 `src/routes/` 作为路由目录，禁止在规则中再引入 `src/router/`
- 本模板固定使用 `src/http/` 作为接口请求目录，禁止在规则中再引入 `src/services/api/`
- 本模板固定使用 `src/views/` 作为页面目录，若某项目实际使用 `src/pages/`，应在该项目自己的扩展规则中声明映射关系，而不是修改模板基线
