# 状态管理规范（NON-NEGOTIABLE）

## 基础规范

- 状态管理必须使用 **Zustand**
- 业务 store 必须放在 `src/stores/modules/`（按业务模块拆分，每个模块一个文件）
- 统一导出入口使用 `src/stores/index.ts`
- **禁止** 在 `src` 下新建其他目录用于存放 store（例如 `src/store`、`src/state` 等）
- Store 文件只负责“纯数据 + 业务动作”，不直接耦合 UI 组件、路由实例、DOM/浏览器 API

## Store 组织规范

- 每个业务模块单独一个 store 文件
- 每个 store 文件必须导出 `useXxxStore` Hook（Hook 与文件语义保持一致）
- Store 中不要直接引用具体 UI 组件，只存纯数据与业务行为
- 推荐在 store 内统一使用 `immer` 中间件，通过 action 更新 state

## 目录结构

- 目标目录结构如下（新增模块请严格遵循）：

```
src/stores/
├── index.ts                 # 统一导出各模块 useXxxStore
└── modules/
    ├── useThemeStore.ts    # useThemeStore
    ├── useUserStore.ts     # useUserStore
    └── useAppStore.ts      # useAppStore
```

## 命名约定

- 文件名：`useXxxStore.ts`（放在 `src/stores/modules/`）
- Hook 名：`useXxxStore`（例如 `useThemeStore`、`useUserStore`、`useAppStore`）
- State 与 Action 需要显式类型化
- State 使用 `interface State` / `type State`
- Action 使用 `type Action`
- Action 命名建议：
- 写状态：`setXxx` / `updateXxx`
- 与 state 字段语义对齐（例如 `setThemeMode` 更新 `themeMode`）

## 持久化（如需要）

当状态需要跨页面或刷新保留时，使用 `zustand/middleware` 的 `persist`。

统一中间件与持久化策略：

- 默认使用 `immer`，用 `set((state) => { ... })` 的方式更新 state
- 需要持久化时必须使用组合：
- `persist`
- `createJSONStorage(() => localStorage)`
- `partialize` 白名单字段策略（只持久化必须的字段，避免无关 state 被长期保存）
- `persist` 的 `name` 统一使用 `app-xxx-store` 形式（例如 `app-theme-store` / `app-user-store` / `app-app-store`）

## 组件使用规范

- 组件中优先使用选择器：`useXxxStore((s) => s.xxx)`，避免一次性订阅整个 store 导致不必要重渲染
- action 调用同样通过选择器获取：`useXxxStore((s) => s.setXxx)` / `useXxxStore((s) => s.updateXxx)`

## 状态边界（NON-NEGOTIABLE）

- 组件私有、短生命周期、仅局部使用的状态，优先使用 `useState` / `useReducer`，不要为此引入全局 store
- 跨页面或跨模块共享的客户端状态，才进入 Zustand store
- 服务端数据与接口缓存默认不进入 Zustand；优先通过路由 `loader`、请求层或专门的数据获取方案管理
- 禁止把“所有接口返回值”机械性塞进全局 store，避免状态源重复与失真
