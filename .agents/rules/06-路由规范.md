# React 路由规范（NON-NEGOTIABLE）

> 推荐基线：React Router 7 + Data Router 风格（`createBrowserRouter` + `RouterProvider`）。
> 本模板固定使用 `src/routes/` 与 `src/views/` 目录，不再在模板层引入 `src/router/` 或 `src/pages/` 双口径。

## 1. 核心目标（NON-NEGOTIABLE）

- **单一事实来源（Single Source of Truth）**：路由表只能在一个入口导出与注册，禁止多处维护。
- **一致性优先**：路径命名、嵌套层级、懒加载、守卫、错误边界必须风格统一。
- **可扩展**：支持布局嵌套、权限控制、错误边界、代码分割、菜单/面包屑元信息。
- **可测试**：守卫与重定向逻辑可被测试覆盖（避免散落在页面 `useEffect`）。

## 2. 推荐模式与禁止事项

- **推荐**：Data Router（`createBrowserRouter`）。使用 `loader/action/errorElement` 组织数据、提交与错误处理。
- **允许**：传统 `<BrowserRouter><Routes>`（存量项目），但禁止与 Data Router “混用两套路由系统”。
- **禁止**：在页面/业务组件内部另起一套 `<Routes>` 作为“子路由系统”（除非明确为独立子应用/微前端，并在架构文档里写清边界）。

## 3. 路由组织与文件结构（建议）

- 路由声明可采用以下之一（团队需固定一种）：
  - **集中单文件**：`src/routes/index.tsx` 维护全部路由对象
  - **按域拆分 + 统一聚合**：`src/routes/modules/*.ts(x)` 导出路由片段，由 `src/routes/index.tsx` 聚合后导出唯一 `routes`
- **页面目录**：`src/views/**`，按业务域分组。
- **路由入口建议**：`src/routes/index.tsx` 只负责创建 router 与导出，避免掺杂业务逻辑。

## 4. 路由对象字段规范（推荐字段集）

- **基础字段**：
  - `path`: string
  - `element`: ReactNode
  - `children`: RouteObject[]
- **Data Router 字段**：
  - `loader`: (args) => data | Response | redirect
  - `action`: (args) => data | Response | redirect
  - `errorElement`: ReactNode（该路由子树的错误边界）
- **元信息（统一放一处）**：
  - 推荐使用 `handle`（React Router 常用约定）或自定义 `meta`，但**团队必须固定一种**。
    典型字段：`title`、`requiresAuth`、`roles`、`breadcrumb`、`menu`、`icon`、`order`。

## 5. 路径命名与参数规范（NON-NEGOTIABLE）

- **静态路径段**：统一 `kebab-case`（例如：`/user-settings`）。
- **动态参数**：`/:userId` 使用 `camelCase`，语义清晰；禁止 `/:id` 这种无语义参数（除非领域模型就叫 id 且团队一致）。
- **资源型路由（推荐）**：
  - 列表：`/users`
  - 详情：`/users/:userId`
  - 编辑：`/users/:userId/edit`
- **禁止**：
  - 新增 `camelCase` 的 path（如 `/themeSetting`）
  - 同一路由同时支持“带尾斜杠/不带尾斜杠”两种形式（需统一一种）

## 6. 嵌套路由与布局（NON-NEGOTIABLE）

- 使用 `children` 表达嵌套结构，父路由 `element` **必须渲染** `<Outlet />`。
- 布局路由（Layout Route）用于承载导航栏/TabBar/侧边栏/容器等公共 UI。
- **禁止**：通过在子页面里手写公共布局来“复制粘贴布局”（会导致状态、样式与行为不一致）。

## 7. 重定向与 404（NON-NEGOTIABLE）

- **默认入口**：`/` 必须显式重定向到首页（如 `/home` 或 `/dashboard`）。
- **404 兜底**：必须提供 `*`（或 `/*`）兜底路由渲染 404 页面。
- **重定向分层**：
  - 结构性重定向（如 `/`、旧路径迁移）写在路由表
  - 权限重定向（未登录/无权限）写在守卫（`loader` 或 wrapper）中
- **禁止**：在多个业务模块重复定义兜底 `*`，避免命中顺序不可控。

## 8. 懒加载与代码分割（NON-NEGOTIABLE）

- **页面级路由必须懒加载**（除极少数关键首屏页，需在 PR 中说明原因）。
- `Suspense fallback` 必须统一（统一 Loading/骨架屏样式与交互）。
- **避免重复 lazy**：同一个页面模块只允许被一个路由 entry 懒加载引用；复用请下沉为组件/容器，而不是“多路由指向同一页面”。

## 9. 鉴权与权限（NON-NEGOTIABLE）

- 守卫实现二选一（团队必须固定一种）：
  - **Data Router 守卫（推荐）**：在受保护路由的 `loader` 中校验，失败则 `redirect('/login?...')`
  - **组件 Wrapper**：`<RequireAuth />` 包裹受保护分支（集中处理跳转与无权限 UI）
- **白名单**：`/login`、`/register`、`/err404` 等公共页必须集中维护白名单。
- **禁止**：页面内 `useEffect` 判断登录再 `navigate`（闪屏、竞态、难测试、逻辑分散）。

## 10. 标题、菜单、面包屑（推荐）

- 标题：从 `handle/meta.title` 统一设置 `document.title`；缺省回退到应用默认标题。
- 菜单/面包屑：建议从路由元信息派生生成（例如 `meta.menu === true`），避免“菜单一份、路由一份”双维护。
- TabBar/SideMenu 的“可见性/排序/图标”统一通过元信息约定字段控制。

## 11. 错误边界与异常处理（NON-NEGOTIABLE）

- 必须有根级 `errorElement`（防止全站白屏）。
- 关键业务子树建议有局部 `errorElement`（局部失败不影响全站）。
- `loader/action` 的异常与返回必须有约定（例如抛 `Response`、抛业务错误类型、或统一错误格式），并在错误页做可读展示。

## 12. 新增/调整路由的落地流程（建议）

- 新建页面组件并导出默认组件（或符合团队约定的组件导出形式）。
- 在“唯一路由表入口”新增路由对象，补齐 `path`、`element`、`handle/meta`。
- 需要鉴权：按团队约定接入守卫（`loader` 或 wrapper），并补齐白名单/权限字段。
- 需要出现在菜单：设置菜单元信息（如 `meta.menu/title/icon/order`）并验证派生结果正确。
- 新增重定向/迁移旧路径：在路由表添加显式 redirect，并补充测试或说明。

## 附：可选增强（从成熟项目实践提炼，非强制）

- **模块化路由片段 + 聚合**：通过 `modules` 拆分减少冲突，聚合后仍保持单一事实来源。
- **统一 LazyLoad 包装**：统一 `Suspense fallback` 与页面切换动画，减少每页重复代码。
- **统一守卫入口**：将标题、埋点、鉴权等横切逻辑收敛到统一入口，便于审计与测试。

## 模板目录基线

推荐结构：

```text
src/
└── routes/
    ├── index.tsx
    ├── guards/
    ├── modules/
    └── utils/
```

- 规则中若出现 `src/router/` 或 `src/pages/`，一律视为历史口径，均应映射为本模板的 `src/routes/` 与 `src/views/`
