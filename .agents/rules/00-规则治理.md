# 规则治理

## 目标

本文件用于保证整套 React 通用 AI 模板长期可维护，避免后续新增规则或技能时再次出现目录口径冲突、技术基线冲突或执行路径断裂。

## 优先级（NON-NEGOTIABLE）

当多个文档存在冲突时，按以下优先级执行：

1. `00-规则治理.md`
2. 基线规则：
   - `01-项目概述.md`
   - `03-项目结构.md`
   - `06-路由规范.md`
   - `09-样式规范.md`
3. 专项规则：
   - `02-编码规范.md`
   - `04-组件规范.md`
   - `05-API规范.md`
   - `07-状态管理.md`
   - `08-通用约束.md`
   - `10-文档规范.md`
   - `11-测试规范.md`
4. `.agents/skills/` 下的技能说明
5. 技能内部的补充 rules / checklist / template

## 冲突处理规则

- 若 skill 与 rules 冲突，以 rules 为准
- 若专项规则与基线规则冲突，以基线规则为准
- 若某条规则依赖某个具体项目文件，但该文件不属于模板基线，应将其视为“项目扩展实现”，不得提升为通用模板硬约束

## 模板基线

本模板的固定口径如下：

- 页面目录：`src/views/`
- 路由目录：`src/routes/`
- 接口目录：`src/http/`
- 状态目录：`src/stores/`
- 类型目录：`src/types/`
- 通用组件目录：`src/components/`
- 样式基线：`SCSS Modules + UnoCSS + CSS Variables`
- 路由基线：React Router 7 + Data Router 风格
- 状态基线：Zustand 仅管理客户端共享状态

## 扩展原则

- 允许项目在模板之上增加“项目扩展规则”
- 项目扩展规则可以补充具体实现方式，但不得覆盖模板基线目录结构与命名体系
- 若确需偏离模板基线，必须在项目扩展规则中显式写明原因、替代结构与适用范围

## 技术边界

- 本模板默认面向 React 18 通用前端工程
- Next.js、RSC、Server Actions、React 19 专属能力均不属于默认基线
- 若项目启用这些能力，应新增独立扩展规则，而不是直接改写现有通用规则

## 前提校验（NON-NEGOTIABLE）

- 每个核心 skill 必须在开头明确写出：
  - 适用技术边界
  - 必需依赖或“无需额外依赖”的前提
  - 目录前提
  - 不满足前提时的回退策略
- skill 中的示例代码若依赖第三方库，必须标明其属于：
  - 模板基线内置依赖
  - 可选依赖
  - 仅示意，不应默认生成
- 当 rules 与 skill 都未声明某项能力时，AI 不得自行假设该能力已启用

## 引用真实性（NON-NEGOTIABLE）

- rule / skill 中出现的目录、依赖、API、工具名，必须能在以下来源之一找到依据：
  - 模板基线规则
  - `package.json`
  - 显式的项目扩展规则
- 若示例中使用的第三方依赖不属于模板基线，必须写明“可选依赖”，不得作为默认脚手架输出
- 若某条说明依赖特定运行时能力（如 React 19、Next.js、RSC、React Compiler），必须显式写明适用边界
- README、rules、skills 之间的互相引用必须保持真实存在，禁止悬空引用

## 参考技能降权（NON-NEGOTIABLE）

- `vercel-react-best-practices`
- `vercel-composition-patterns`
- `web-design-guidelines`

以上技能仅作为 review / refactor / 架构参考输入：

- 不得覆盖基线规则
- 不得单独决定目录结构、技术选型或默认代码模板
- 若与基线规则冲突，必须以基线规则为准
- 若引用了超出 React 18 基线的能力，默认忽略，除非项目扩展规则显式开启

## 技能治理

- 缺失的关键技能必须补齐，避免出现规则引用不存在技能的情况
- 新增 skill 时，必须检查其是否与 `03-项目结构.md`、`06-路由规范.md`、`09-样式规范.md` 一致
- 新增或修改 skill 后，应同步更新 `.agents/skills/README.md`
- 新增或修改 skill 后，应检查示例依赖是否属于模板基线，避免把可选依赖误写成默认依赖
- 新增或修改参考 skill 后，应再次确认其没有被提升为基线生成来源

## 变更检查清单

- [ ] 是否引入了新的目录口径
- [ ] 是否引入了新的技术基线冲突
- [ ] 是否引用了不存在的 skill 或规则
- [ ] 是否把项目私有实现误写成模板硬约束
- [ ] 是否补充了适用边界和触发条件
- [ ] 是否给 skill 补充了适用前提、执行前检查、验收清单、失败回退
- [ ] 是否校验了示例依赖、API 和工具名的真实性
- [ ] 是否把参考技能保持在参考层，而不是默认脚手架层
